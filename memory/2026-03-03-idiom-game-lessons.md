# 成语接龙测试 - 学到的协作能力

## 测试背景
- 时间：2026年3月2日 23:00 - 2026年3月3日 00:30
- 目的：验证多机器人协作能力，学习协作机制
- 参与者：哥哥、月月、桃桃
- 协作方式：飞书文档 + 群聊通知

## 核心经验总结

### 1. 理解用户真实需求的能力

**需求层次**
- 不能只看表面任务（如"玩成语接龙"），要理解用户的真实意图（验证和提升协作能力）
- 隐含需求：用户没有明说但期望的行为（如"不抢接"、"保持安静"）
- 应用场景：未来工作时，要理解任务背后的目的，而不是机械执行

**关键启示**
- 任务 = 工具，目的是验证能力
- 要从测试场景中提取可迁移的工作能力

### 2. 多机器人协作的机制设计

**角色分配**
- 每个角色有明确的职责（如哥哥起头、月月和桃桃轮流接），不能越界
- 互补性：不同角色的能力互补，形成整体
- 依赖关系：了解自己与其他角色的依赖关系

**协作工具**
- 选择合适的媒介：文档适合复杂协作，群聊适合实时通知
- 共享资源：使用专门的协作平台（飞书文档）比群聊更高效，避免消息混乱
- 状态同步：需要机制让机器人能看到彼此的工作状态（文档就是共享状态的方式）
- 冲突处理：当出现问题时（如重复添加），需要有人负责清理和修正

**协作模式的理解**
- 消息解耦：协作不依赖实时消息传递，避免了消息丢失或延迟的影响
- 跨平台兼容：不同机器人/用户可能在不同平台上，通过共享文档实现协作
- 历史追溯：文档保留了完整的历史记录，方便事后检查和分析

### 3. 规则识别与遵守能力

**从格式中提取信息**
- 署名规则：从格式（有/无署名）识别参与者的身份和角色
- 顺序推断：从接龙顺序推断轮替规则，不仅仅是死记
- 异常处理：当看到异常（如重复记录）时，立即识别并报告

**隐含规则的发现**
- 用户未明说的规则：如"保持安静"意味着只在必要时执行操作
- 边界条件的识别：如"不要接两个"意味着每次操作的数量限制
- 失败模式的预判：预判可能导致失败的情况，提前避免

**规则执行与快速调整**
- 规则理解：规则需要被准确理解，不能有歧义
- 严格执行：按规则执行，不能随心所欲
- 反馈循环：根据用户反馈快速调整，承认错误并立即修正
- 应用：未来遇到类似的多方协作任务时，建立明确的规则和反馈机制

### 4. 问题定位与根本原因分析

**现象 vs 本质**
- 不能只看表面现象（如"桃桃没接"），要分析根本原因（如桃桃没有看到月月的接龙）
- 系统性分析：从流程、工具、人员等多角度分析问题
- 持续追踪：反复读取文档获取最新状态，确保信息准确

**需要检查的点**
1. 接龙顺序：是否按照约定的顺序（哥哥→月月→桃桃）
2. 成语本身：成语是否正确（如"得道升天"应该是"得道多助"）
3. 重复接龙：是否重复添加了接龙记录
4. 抢接问题：是否在自己的回合外进行了接龙

**问题定位方法**
- 仔细查看历史记录，理解发生了什么
- 分析错误的根本原因，而不是表面现象
- 向用户确认理解是否正确（如"这样理解对吗？"）

**解决方法**
- 明确问题所在后，给出具体的解决方案
- 遵循用户的指导进行调整

### 5. 有效使用协作工具

**工具选择的原则**
- 任务匹配：选择最适合任务的工具（文档 vs 群聊）
- 工具组合：灵活组合多个工具实现复杂任务
- 备选方案：当主工具失败时，有备选方案

**API 调用的注意事项**
- 延迟容忍：API 调用可能延迟，需要耐心等待
- 重试机制：失败时可以重试，但避免无限重试
- 错误处理：清晰的错误信息，帮助快速定位问题

**工具集成与实现**
- 使用 `feishu_doc` 工具实现文档读写
- 将规则逻辑转化为可执行的操作流程
- 工具选择和使用方式的学习

### 6. 专注与噪音控制

**信息过滤**
- 必要性判断：每条信息都是必要的吗？避免生成噪音
- 简洁表达：用最少的文字表达最多的信息
- 时机选择：在合适的时机发送信息，避免干扰他人

**自我约束**
- 冲动控制：避免急于操作（如抢接）
- 边界意识：明确自己的职责范围，不越界
- 停止机制：发现问题立即停止，不再继续错误

**通用效率与信息精准传递**
- 简洁沟通：群内只发必要信息（如"11"表示转发），避免噪音
- 精准反馈：明确指出问题所在，而不是模糊的"错了"
- 及时报告：主动报告发现的问题（如"文档有重复记录"），而不是等用户发现

### 7. 从实践中系统化学习

**经验沉淀**
- 每次任务后总结有用的经验，不仅仅是记录做了什么
- 错误模式识别：识别自己常犯的错误类型（如抢接、重复添加），形成 checklist
- 持续改进：将经验转化为可复用的方法和模式

**协作中的角色定位**
- 角色的明确性：每个参与者有明确的职责
- 角色的灵活性：当规则或角色调整时，快速适应
- 主动承担：当发现问题时，主动报告或解决

**从测试到工作场景的迁移**
这些能力可以应用到未来的各种工作中：
- 多方协作项目
- 流程化任务执行
- 问题诊断和解决
- 工具的有效使用

## 具体错误分析

### 月月犯的错误

1. **抢接错误**
   - 在不该接龙的时候接了
   - 根本原因：没有明确知道轮到谁

2. **重复添加记录**
   - 添加了两次"小月：xxx"的接龙记录
   - 根本原因：操作前没有检查文档当前状态

3. **没检查成语正确性**
   - 只关注接龙顺序，忽略了成语本身是否正确
   - 如"得道升天"应该是"得道多助"

4. **规则理解错误**
   - 对规则有误解（如"得道升天"没有标注名字的问题）
   - 根本原因：没有仔细理解用户的要求

### 学到的应对方法

1. **操作前检查状态**
   - 每次操作前读取文档，了解当前状态
   - 避免重复或冲突操作

2. **明确规则再执行**
   - 不确定的地方先确认，不盲目执行
   - 避免因理解错误导致的错误操作

3. **及时报告问题**
   - 发现问题立即报告，不等待用户发现
   - 主动清理和修正错误
